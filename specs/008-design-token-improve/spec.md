# Feature Specification: 디자인 토큰 시스템 개선

**Feature Branch**: `008-design-token-improve`  
**Created**: 2026-04-06  
**Status**: Draft  
**Input**: User description: "improve design token. brand color is `#d92b33`, but problem is that color is looks similar with destructive color code. we need good color scale. we will make most korean sites so need common korean font family. optional: we need dark mode"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 브랜드 컬러 스케일 적용 (Priority: P1)

디자이너 또는 개발자가 브랜드 컬러(`#d92b33`)를 UI에 적용할 때, 오류/파괴적 동작을 나타내는 색상과 시각적으로 명확히 구분되는 컬러 스케일을 사용한다. 브랜드 컬러는 고유한 10단계 스케일(50~950)로 정의되며, 파괴적 컬러는 별도의 색상 계열로 분리된다.

**Why this priority**: 브랜드 색상이 오류 색상과 혼동되면 사용자가 UI 상태를 잘못 해석하게 되어 신뢰도와 사용성이 저하된다. 이것이 이번 개선의 핵심 문제이므로 P1이다.

**Independent Test**: 브랜드 컬러 스케일과 파괴적 컬러 스케일을 나란히 시각화하여 색상 차이가 명확히 보이는지 확인하는 것만으로 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 브랜드 컬러 `#d92b33`이 정의된 디자인 토큰 시스템이 있을 때, **When** 브랜드 컬러와 파괴적 컬러를 동시에 화면에 표시하면, **Then** 두 색상은 육안으로 명확히 구분되어야 한다 (색상 이론 기준 충분한 색조 차이).
2. **Given** 디자인 토큰 문서가 있을 때, **When** 브랜드 컬러 스케일 50~950 단계를 확인하면, **Then** 각 단계가 명도 및 채도 기준으로 일관된 간격을 유지해야 한다.
3. **Given** 파괴적 컬러 토큰이 정의되어 있을 때, **When** 파괴적 컬러를 확인하면, **Then** 브랜드 컬러(`#d92b33`)와 색조(hue) 차이가 15° 이상이어야 한다.

### User Story 2 - 한국어 폰트 패밀리 적용 (Priority: P2)

한국어 콘텐츠를 주로 제공하는 사이트를 구축할 때, 디자이너 또는 개발자가 가독성이 높고 한국에서 널리 사용되는 폰트 패밀리를 토큰에서 바로 참조하여 사용한다.

**Why this priority**: 한국어 사이트 특화가 명시된 요구사항이며, 적절한 폰트 없이는 한글 가독성이 저하된다. 브랜드 컬러 구분 다음으로 중요한 사용자 경험 요소다.

**Independent Test**: 폰트 토큰을 참조하여 한국어 텍스트를 렌더링하고 가독성과 스타일 일관성을 확인하는 것으로 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 폰트 토큰이 정의된 시스템이 있을 때, **When** 한국어 텍스트를 렌더링하면, **Then** 한국어에 최적화된 폰트(Pretendard 또는 Noto Sans KR)로 표시되어야 한다.
2. **Given** 특정 폰트가 로드되지 않은 환경에서, **When** 폰트 토큰을 사용하면, **Then** 시스템 한국어 폰트(Apple SD Gothic Neo, Malgun Gothic 등)로 graceful fallback이 적용되어야 한다.
3. **Given** 영문 텍스트와 한국어 텍스트가 혼재할 때, **When** 동일한 폰트 토큰을 사용하면, **Then** 두 언어 모두 자연스럽고 가독성 있게 표시되어야 한다.

---

### User Story 3 - 다크 모드 지원 (Priority: P3)

사용자가 다크 모드를 선호하는 환경에서 사이트에 접근할 때, UI 전체가 다크 모드에 맞는 색상 토큰으로 자동 전환되어 눈의 피로를 줄이고 일관된 경험을 제공한다.

**Why this priority**: 선택적(OPTIONAL) 요구사항으로, 브랜드 컬러와 폰트 개선이 완료된 이후 추가 가치를 제공한다. 현대 한국 웹 사이트에서 다크 모드는 점점 표준이 되고 있어 P3로 포함한다.

**Independent Test**: 다크 모드로 전환했을 때 모든 의미론적 색상 토큰이 다크 모드용 값으로 변경되고 가독성이 유지되는지 확인하는 것으로 독립 검증 가능하다.

**Acceptance Scenarios**:

1. **Given** 다크 모드 설정이 활성화된 환경에서, **When** 사이트를 로드하면, **Then** 모든 배경, 텍스트, 경계선 색상이 다크 모드에 적합한 토큰 값으로 표시되어야 한다.
2. **Given** 다크 모드에서 브랜드 컬러가 사용될 때, **When** 화면을 확인하면, **Then** 브랜드 컬러는 파괴적 컬러와 다크 모드에서도 명확히 구분되어야 한다.
3. **Given** 라이트/다크 모드 전환이 가능한 환경에서, **When** 모드를 전환하면, **Then** 페이지 새로고침 없이 색상이 즉시 전환되어야 한다.

---

### Edge Cases

- 브랜드 컬러 스케일의 중간 단계(예: 500)가 정확히 원본 `#d92b33`과 일치하지 않을 때 어떻게 처리하는가?
- 사용자의 운영체제가 한국어 폰트를 지원하지 않는 환경(예: 일부 Linux)에서 폰트 fallback이 어떻게 동작하는가?
- 다크 모드에서 접근성 기준(WCAG AA, 명암비 4.5:1 이상)을 충족하지 못하는 색상 조합이 발생하면 어떻게 처리하는가?
- 파괴적 컬러와 브랜드 컬러가 색맹 사용자에게 구분 가능한가?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 토큰 시스템은 브랜드 컬러 `#d92b33`을 기준으로 50~950 사이의 10단계 색상 스케일을 제공해야 한다.
- **FR-002**: 파괴적(오류/위험) 색상 토큰은 브랜드 컬러와 명확히 구분되는 별도의 색조(hue) 계열로 정의되어야 한다.
- **FR-003**: 의미론적 색상 토큰(브랜드, 파괴적, 중립, 성공, 경고 등)은 원시 팔레트와 분리된 별도 레이어로 정의되어야 한다.
- **FR-004**: 폰트 패밀리 토큰은 Pretendard를 1순위로, Noto Sans KR을 2순위로, 시스템 한국어 폰트를 최종 fallback으로 포함해야 한다.
- **FR-005**: 폰트 토큰은 본문(body), 제목(heading), 모노스페이스(mono) 등 용도별 폰트 패밀리를 구분하여 제공해야 한다.
- **FR-006**: 다크 모드 지원 시, 모든 의미론적 색상 토큰은 라이트/다크 모드 각각의 값을 가져야 한다.
- **FR-007**: 다크 모드 전환은 시스템 설정(OS 레벨 prefers-color-scheme) 및 수동 전환 방식 모두 지원해야 하며, 수동 전환 시 사용자의 선택은 브라우저 로컬 저장소에 유지되어 재방문 시에도 적용되어야 한다. 로컬 저장소 접근 실패 시에도 DOM 상태(`data-theme` 속성) 적용은 정상 동작해야 한다 (graceful degradation).
- **FR-008**: 모든 색상 토큰 조합(배경+텍스트)은 WCAG AA 기준(명암비 4.5:1) 이상을 만족해야 한다.
- **FR-009**: 브랜드 컬러와 파괴적 컬러는 색맹 시뮬레이션(적록색맹 기준)에서도 구분 가능해야 한다.
- **FR-010**: 토큰 시스템은 CSS Custom Properties와 TypeScript 객체 두 가지 형식으로 동시에 출력되어야 하며, 두 출력 간 값이 동기화 상태를 유지해야 한다.

### Key Entities

- **브랜드 컬러 팔레트**: `#d92b33`을 중심으로 생성된 10단계 원시 색상 스케일. 각 단계는 명도/채도 기준으로 일관된 간격을 가짐.
- **파괴적 컬러 팔레트**: 브랜드 컬러와 색조 차이가 충분한(≥15°) 오류/위험 전용 색상 스케일.
- **의미론적 색상 토큰**: 원시 팔레트를 참조하는 역할 기반 토큰 레이어 (예: `color.brand.primary`, `color.destructive.default`).
- **폰트 패밀리 토큰**: 용도별 폰트 패밀리 정의. 한국어 최적화 폰트 우선 순위 포함.
- **테마 토큰**: 라이트/다크 모드별 의미론적 토큰 값 매핑.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 브랜드 컬러와 파괴적 컬러를 나란히 배치했을 때 색조(hue) 차이가 15° 이상이어 육안으로 즉시 구분 가능하다.
- **SC-002**: 브랜드 컬러 스케일 10단계가 완전히 정의되어, 디자이너가 어두운 변형(hover, pressed)과 밝은 변형(tint, background)을 별도 계산 없이 바로 참조할 수 있다.
- **SC-003**: 한국어 텍스트 렌더링 시 Pretendard 또는 Noto Sans KR이 로드되며, 폰트 누락 시 시스템 폰트로 자연스럽게 fallback된다.
- **SC-004**: 모든 텍스트-배경 색상 토큰 조합이 WCAG AA(4.5:1) 이상 명암비를 만족한다.
- **SC-005**: (다크 모드) 다크 모드 전환 후 모든 UI 컴포넌트가 지정된 다크 모드 토큰 값으로 표시되며, 라이트 모드 토큰이 잔류하지 않는다.
- **SC-006**: 브랜드 컬러와 파괴적 컬러의 색조(hue) 차이가 자동화 계약 테스트로 검증된다 (≥15° 필수, 실제 구현 35°). 이 hue 차이는 적록색맹 환경에서도 명도/채도 차이를 통해 두 색상을 구분 가능하게 보장한다.

## Clarifications

### Session 2026-04-06

- Q: 토큰 출력 포맷은 무엇인가? → A: CSS Custom Properties + TypeScript 객체 이중 출력
- Q: 폰트 로딩 방식은 무엇인가? → A: npm 패키지 self-hosted
- Q: 다크 모드 수동 전환 설정을 유지하는가? → A: localStorage에 유지 (재방문 시에도 기억)
- Q: 브랜드 컬러 `#d92b33`의 스케일 앵커 위치는? → A: 600 단계 앵커 (실제 명도 기반)

## Assumptions

- Pretendard와 Noto Sans KR은 npm 패키지로 self-hosted 방식으로 로드된다. 외부 CDN을 사용하지 않아 GDPR·개인정보 이슈를 방지하며, 두 폰트 모두 SIL OFL 1.1 라이선스임을 가정한다.
- 브랜드 컬러 `#d92b33`은 변경 불가한 확정값이며, 색상 스케일은 이 값을 600 단계 앵커로 고정하여 생성된다. 이에 따라 50~500 범위에 밝은 tint/배경 색상이, 700~950 범위에 어두운 hover/pressed 변형이 배치된다.
- 파괴적 컬러는 오렌지-레드 계열(hue ~10~20°) 또는 순수 오렌지 계열로 분리하여 브랜드 레드와 구분한다.
- 다크 모드는 선택적 요구사항으로, 라이트 모드가 완전히 구현된 이후 추가한다.
- 현재 토큰 시스템은 `packages/tokens` 패키지에 존재하며, 기존 토큰 구조를 확장하는 방식으로 진행한다.
- WCAG AA(4.5:1) 기준을 최소 준수하며, AAA(7:1)는 권장사항으로만 적용한다.
- 파괴적 컬러의 전경색(`destructiveForeground`)은 흰색 대신 짙은 회색(`grayHsl['950']`)을 사용한다. 오렌지-레드 배경 위에서 짙은 회색이 흰색보다 WCAG 명암비 여유가 크기 때문이다.
