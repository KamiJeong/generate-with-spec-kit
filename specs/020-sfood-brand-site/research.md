# Phase 0 연구: SFood 브랜드 사이트

## 결정 1: 앱 위치는 `apps/sfood`로 고정한다

**결정**: 신규 브랜드 사이트는 `C:\Users\USER\Documents\GitHub\generate-with-spec-kit\apps\sfood`에 생성한다.

**근거**: 사용자 요구가 `/apps/sfood`를 명시했고, `pnpm-workspace.yaml`이 이미 `apps/*`를 workspace 패턴으로 포함한다. 기존 `packages/web`는 재사용 가능한 패턴 참고용이며, `packages/sfood`는 소스 없이 `node_modules`만 있어 기능 구현 위치로 적합하지 않다.

**검토한 대안**:

- `packages/sfood`: 현재 소스/패키지 메타데이터가 없고 사용자 요청 경로와 다르다.
- `packages/web` 확장: 기존 소개 사이트와 SFood 브랜드 사이트의 목적이 달라 범위가 섞인다.
- `app/sfood-brand`: workspace에 포함되지 않은 단수 `app` 경로이며 생성 산출물 중심이라 새 기능 위치로 부적합하다.

## 결정 2: 정적 React/Vite 앱으로 구현한다

**결정**: Vite, React, TypeScript, Tailwind CSS 4 기반의 정적 프런트엔드 앱으로 구현한다.

**근거**: 명세가 API 없음, 템플릿성 콘텐츠, 공개 페이지 8개를 요구한다. 기존 `packages/web`가 동일 계열 스택을 사용하므로 repo 패턴을 따를 수 있고, `@myorg/ui`와 `@myorg/tokens`를 직접 재사용할 수 있다.

**검토한 대안**:

- 서버 렌더링 앱: API와 서버 운영 범위가 없으므로 과하다.
- 순수 HTML/CSS: 디자인 시스템 컴포넌트와 테스트 재사용성이 낮다.
- 기존 `packages/web`에 라우트 추가: SFood 앱 독립성이 떨어지고 사용자 요청 경로를 만족하지 못한다.

## 결정 3: 새 라우팅 런타임 의존성을 추가하지 않는다

**결정**: 8개 정적 URL은 `window.location.pathname`, `history.pushState`, typed route map 기반의 단순 클라이언트 라우팅으로 처리한다.

**근거**: 페이지 수가 작고 상세/동적 라우트가 없다. 헌법의 단순성 원칙과 외부 의존성 제한에 따라 React Router 같은 새 런타임 의존성을 추가하지 않는 편이 적합하다.

**검토한 대안**:

- React Router 추가: 기능 요구 대비 의존성 증가가 크다.
- 해시 라우팅: 요청 URL(`/about`, `/support/news`)과 맞지 않는다.
- 다중 HTML 파일: 공통 레이아웃과 테스트 중복이 커진다.

## 결정 4: 콘텐츠는 typed constant로 관리한다

**결정**: 페이지, 브랜드 라인, 공지/뉴스 항목, FAQ, 채용 단계는 `src/content/site.ts`의 타입 있는 상수로 관리한다.

**근거**: API와 CMS가 범위 밖이며, TypeScript 타입으로 필수 필드 누락을 조기에 잡을 수 있다. 콘텐츠 검토 테스트에서 금지어, URL, 요약 존재 여부를 검증하기 쉽다.

**검토한 대안**:

- JSON 파일: 타입 보강이 별도로 필요하고 아이콘/컴포넌트 참조가 불편하다.
- Markdown 파일: 현재 범위에는 파서와 로더가 불필요하다.
- 하드코딩된 JSX 카피: 재사용과 테스트가 어렵다.

## 결정 5: Storybook 레퍼런스와 디자인 시스템 컴포넌트를 우선 사용한다

**결정**: 레이아웃은 Storybook의 `BrandSite` 패턴을 참고하고, 내비게이션은 `NavigationMenu`, 콘텐츠 묶음은 `Card`, FAQ는 `Accordion`, 주요 행동은 `Button`을 우선 사용한다.

**근거**: `DESIGN.md`가 semantic token과 시스템 컴포넌트 우선을 요구한다. Storybook 문서는 해당 컴포넌트의 실제 사용 패턴을 제공하므로 접근성, 반응형, 상태 표현을 repo 관례에 맞출 수 있다.

**검토한 대안**:

- 전부 커스텀 HTML로 작성: 빠르지만 디자인 시스템 일관성이 떨어진다.
- 새 컴포넌트 라이브러리 추가: 헌법의 의존성/단순성 원칙에 위배된다.
- 랜딩 페이지용 외부 템플릿 사용: 브랜드 요구와 토큰 규칙을 보장하기 어렵다.

## 결정 6: 품질 신뢰 표현은 일반화하고 금지어 테스트로 보호한다

**결정**: 콘텐츠에 정확한 인증/수상 명칭을 넣지 않고, 테스트에서 `FSSC 22000`, `HACCP`, `DLG` 같은 금지어가 없는지 확인한다.

**근거**: 명확화 결과가 정확한 실명 주장 대신 일반화된 품질 신뢰 표현을 요구한다. 테스트로 보호하면 구현 중 실수로 실제 명칭을 추가하는 일을 줄일 수 있다.

**검토한 대안**:

- 정확한 명칭을 샘플 문구로 사용: 법적/브랜드 리스크가 있다.
- 품질 표현을 모두 제거: 명세의 신뢰 요소 요구를 약화한다.
- 리뷰만으로 확인: 자동 회귀 방지가 어렵다.
