# Feature Specification: AI Wiki Portal Platform 화면 개발

**Feature Branch**: `021-ai-wiki-portal`  
**Created**: 2026-04-15  
**Status**: Draft  
**Input**: User description: "[Blueprint] AI Wiki Portal Platform 화면 개발 (by claude)"

## 개요

LLM Wiki 기반 AI 개발 가이드 플랫폼 화면으로, 비개발자도 AI를 활용하여 서비스 개발을 수행할 수 있도록 지원한다. `DocsHubLayout` 기반의 업무용 Admin 디자인을 채용하며, 모든 데이터는 Mock으로 구현한다. 앱 위치: `/apps/ai-wiki-portal-platform-by-claude`

---

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 플랫폼 홈 & 프로젝트 대시보드 조회 (Priority: P1)

비개발자 직원이 플랫폼에 접속하여 현재 진행 중인 프로젝트 현황과 자신의 작업 상태를 한눈에 파악한다.

**Why this priority**: 플랫폼의 진입점이자, 조직 내 개발 프로젝트 가시성을 즉시 제공하는 핵심 기능이다. 이 화면 하나로 MVP의 가치를 전달할 수 있다.

**Independent Test**: 대시보드 화면을 열고 Mock 데이터 기반의 프로젝트 카드, 진행률, 참여자 현황이 올바르게 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 플랫폼에 접속했을 때, **When** 홈(대시보드) 화면을 열면, **Then** 진행 중인 프로젝트 목록(카드 형태), 전체 진행률, 최근 활동 피드가 표시된다.
2. **Given** 프로젝트 목록이 표시되어 있을 때, **When** 특정 프로젝트 카드를 클릭하면, **Then** 해당 프로젝트의 상세 페이지로 이동한다.
3. **Given** 대시보드가 표시될 때, **When** 사용자별 작업 상태 섹션을 확인하면, **Then** 담당자, 진행 단계, 상태 배지(진행중/완료/블로킹)가 표시된다.

---

### User Story 2 - Wiki 문서 탐색 및 조회 (Priority: P2)

비개발자 직원이 좌측 사이드바 네비게이션을 통해 가이드 문서를 탐색하고, 선택한 문서의 내용을 메인 영역에서 읽는다.

**Why this priority**: 플랫폼의 핵심은 실행 가능한 Wiki 문서 제공이며, 문서 탐색 없이는 AI 가이드 기능도 의미가 없다.

**Independent Test**: 좌측 트리 메뉴에서 문서를 선택하면 해당 문서 내용(Mock)이 메인 영역에 렌더링되고, 목차(TOC)가 우측에 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 사용자가 Wiki 섹션에 있을 때, **When** 좌측 사이드바에서 카테고리를 펼치면, **Then** 하위 문서 목록이 트리 형태로 표시된다.
2. **Given** 문서 목록이 표시될 때, **When** 특정 문서를 클릭하면, **Then** 해당 문서 내용이 메인 영역에 마크다운 형태로 렌더링된다.
3. **Given** 문서가 표시될 때, **When** 우측 패널을 확인하면, **Then** 문서 내 헤딩 기반의 목차(TOC)가 표시되어 섹션 이동이 가능하다.
4. **Given** 검색창이 있을 때, **When** 키워드를 입력하면, **Then** 관련 문서 제목/내용 기반의 검색 결과가 표시된다.

### User Story 3 - Blueprint 생성 화면 (Priority: P2)

비개발자 직원이 자연어로 서비스 요구사항을 입력하면, Blueprint가 생성되어 PRD, 아키텍처, 개발 환경 가이드 등이 자동으로 제시된다.

**Why this priority**: Blueprint는 플랫폼의 핵심 자동화 기능으로, 비개발자가 개발 프로세스를 시작하는 출발점이다.

**Independent Test**: 요구사항 입력 폼에 텍스트를 입력하고 "생성" 버튼을 클릭하면, Mock Blueprint 결과(PRD 섹션, 아키텍처 다이어그램 텍스트, 환경 가이드 목록)가 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** Blueprint 생성 페이지에 있을 때, **When** 요구사항 텍스트를 입력하고 생성을 요청하면, **Then** 로딩 인디케이터가 표시된 후 Blueprint 결과물이 렌더링된다.
2. **Given** Blueprint 결과가 표시될 때, **When** 각 섹션(PRD, 아키텍처, 환경 가이드, API 설계)을 확인하면, **Then** 탭 또는 아코디언 형태로 섹션별 내용이 구분되어 표시된다.
3. **Given** Blueprint가 생성된 후, **When** "프로젝트 시작" 버튼을 클릭하면, **Then** 해당 Blueprint 기반의 새 프로젝트가 대시보드에 등록된다(Mock).

### User Story 4 - Get Started 가이드 단계별 진행 (Priority: P3)

비개발자 직원이 개발 환경 구성 가이드를 단계별로 따라가며, 각 단계의 완료 여부를 체크하고 AI 도움말을 요청한다.

**Why this priority**: 단계별 실행 가이드는 비개발자의 실제 진행을 돕는 실행 중심 기능이다.

**Independent Test**: Get Started 페이지에서 단계 목록이 표시되고, 각 단계를 완료 체크하면 진행률이 업데이트되며, "AI 도움" 버튼 클릭 시 안내 메시지가 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** Get Started 페이지에 있을 때, **When** 화면을 열면, **Then** 번호가 매겨진 단계 목록과 각 단계의 설명, 예상 소요 시간이 표시된다.
2. **Given** 단계 목록이 표시될 때, **When** 특정 단계의 체크박스를 클릭하면, **Then** 해당 단계가 완료 상태로 표시되고 전체 진행률(%)이 업데이트된다.
3. **Given** 특정 단계를 진행 중일 때, **When** "AI 도움말" 버튼을 클릭하면, **Then** 해당 단계에 대한 AI 안내 메시지 패널이 슬라이드 인으로 표시된다.

---

### User Story 5 - 프로젝트 상세 및 GitHub 연동 현황 (Priority: P3)

팀 관리자가 특정 프로젝트의 상세 현황 — 진행 단계, 참여자, GitHub 커밋 이력, 이슈 목록 — 을 확인한다.

**Why this priority**: 프로젝트 가시성과 협업 강화를 위한 기능으로, 조직 내 공유 목적을 달성한다.

**Independent Test**: 프로젝트 상세 페이지에서 Mock GitHub 연동 데이터(커밋 수, 이슈 수, 브랜치 현황)와 참여자 목록이 표시되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 프로젝트 상세 페이지에 있을 때, **When** 화면을 열면, **Then** 프로젝트 메타 정보(이름, 설명, 상태), 진행 단계 타임라인, 참여자 목록이 표시된다.
2. **Given** GitHub 연동 섹션이 있을 때, **When** 확인하면, **Then** 최근 커밋 목록, 오픈 이슈 수, 브랜치 현황이 Mock 데이터로 표시된다.
3. **Given** 문제 발생 시, **When** "AI에게 질문" 버튼을 클릭하면, **Then** AI 질의 패널이 열리고 입력창이 포커스된다.

---

### Edge Cases

- 프로젝트가 없을 때: 빈 상태(Empty State) UI가 "아직 프로젝트가 없습니다. Blueprint를 생성하여 시작하세요" 메시지와 CTA 버튼으로 표시된다.
- 문서 검색 결과가 없을 때: "검색 결과가 없습니다" 메시지와 검색어 초기화 버튼이 표시된다.
- Blueprint 생성 중 오류 발생 시: 에러 메시지와 재시도 버튼이 표시된다.
- 모바일/태블릿 화면에서: 좌측 사이드바가 오버레이 드로어 형태로 전환된다. (단, v1 범위는 데스크탑 우선)
- 진행 단계가 100% 완료되었을 때: 완료 배너와 다음 단계 안내가 표시된다.

## Requirements *(mandatory)*

### Functional Requirements

**레이아웃 & 네비게이션**
- **FR-001**: 시스템은 DocsHubLayout 기반의 3단 레이아웃(좌측 사이드바 네비게이션 + 메인 콘텐츠 영역 + 선택적 우측 패널)을 제공해야 한다.
- **FR-002**: 좌측 사이드바는 접기/펼치기(collapse/expand) 기능을 지원해야 한다.
- **FR-003**: 상단 헤더는 전역 검색창, 알림 아이콘, 사용자 프로필 메뉴를 포함해야 한다.
- **FR-004**: 현재 활성화된 메뉴 항목은 시각적으로 강조(활성 상태)되어야 한다.

**대시보드**
- **FR-005**: 대시보드는 진행 중인 프로젝트 카드 목록, 전체 통계 요약(총 프로젝트 수, 완료율, 활성 사용자 수), 최근 활동 피드를 표시해야 한다.
- **FR-006**: 프로젝트 카드는 이름, 설명, 진행률 바, 상태 배지, 참여자 아바타를 포함해야 한다.
- **FR-007**: 사용자는 프로젝트 카드를 클릭하여 상세 페이지로 이동할 수 있어야 한다.
- **FR-024**: 앱은 URL 기반 클라이언트 사이드 라우팅을 사용하며, 각 화면은 고유 경로를 가진다 (`/dashboard`, `/wiki`, `/wiki/:docId`, `/blueprint`, `/get-started`, `/projects/:id`). 브라우저 뒤로가기 및 직접 URL 접근이 지원되어야 한다.

**Wiki 문서 탐색**
- **FR-008**: 좌측 사이드바는 카테고리별 문서를 트리 구조로 탐색할 수 있어야 한다.
- **FR-009**: 메인 영역에서 선택된 문서의 내용이 렌더링되어야 한다(Mock 마크다운 콘텐츠).
- **FR-010**: 우측 패널에 현재 문서의 목차(TOC)가 표시되어야 한다.
- **FR-011**: 전역 검색은 Wiki 문서(제목 및 내용)만을 대상으로 하며, 키워드 일치 결과를 문서 목록 형태로 표시해야 한다. 프로젝트 및 Blueprint는 검색 범위에서 제외된다.

**Blueprint 생성**
- **FR-012**: Blueprint 생성 페이지는 요구사항 입력 텍스트 에어리어, 생성 버튼, 결과 표시 영역을 포함해야 한다.
- **FR-013**: 생성 버튼 클릭 시 로딩 상태가 표시되고, 완료 후 Mock Blueprint 결과(PRD, 아키텍처, 환경 가이드, API 설계 섹션)가 렌더링되어야 한다.
- **FR-014**: Blueprint 결과는 탭 또는 아코디언 형태로 섹션을 구분하여 표시해야 한다.
- **FR-015**: "프로젝트 시작" 버튼 클릭 시 새 프로젝트가 대시보드에 Mock 등록되어야 한다.

**Get Started 가이드**
- **FR-016**: Get Started 페이지는 번호가 매겨진 단계 목록을 표시하고, 각 단계에 완료 체크박스를 제공해야 한다.
- **FR-017**: 단계 완료 시 진행률(%)이 실시간으로 업데이트되어야 한다.
- **FR-018**: 각 단계에 "AI 도움말" 버튼이 있어, 클릭 시 해당 단계의 안내 패널이 표시되어야 한다.

**프로젝트 상세**
- **FR-019**: 프로젝트 상세 페이지는 메타 정보, 진행 단계 타임라인, 참여자 목록, GitHub 연동 현황(Mock)을 표시해야 한다.
- **FR-020**: "AI에게 질문" 버튼을 통해 AI 질의 패널을 열 수 있어야 한다.

**공통**
- **FR-021**: 모든 데이터는 Mock 데이터로 구현되며, API 연동은 범위 외이다.
- **FR-022**: 앱은 `/apps/ai-wiki-portal-platform-by-claude` 경로에 독립적으로 구성되어야 한다.
- **FR-023**: 기존 `@myorg/ui` 컴포넌트 라이브러리를 활용하여 UI를 구성해야 한다.

### Key Entities

- **프로젝트(Project)**: 이름, 설명, 상태(진행중/완료/블로킹), 진행률(%), 참여자 목록, GitHub 저장소 URL, 생성일
- **문서(Document)**: 제목, 카테고리, 내용(마크다운), 최종 수정일, 작성자
- **Blueprint**: 입력 요구사항 텍스트, PRD 내용, 아키텍처 내용, 환경 가이드 내용, API 설계 내용, 생성일
- **사용자(User)**: 이름, 아바타 URL, 역할(관리자/일반), 담당 프로젝트 목록
- **진행 단계(Step)**: 번호, 제목, 설명, 완료 여부, 예상 소요 시간, AI 도움말 내용
- **활동 피드(ActivityFeed)**: 이벤트 유형, 사용자, 프로젝트 참조, 발생 시각

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 비개발자 사용자가 플랫폼에 접속하여 대시보드에서 원하는 프로젝트를 찾는 데 1분 이내에 완료할 수 있다.
- **SC-002**: Wiki 문서 탐색 시, 3번 이내의 클릭으로 원하는 문서에 도달할 수 있다.
- **SC-003**: Blueprint 생성 화면에서 요구사항 입력 후 결과 확인까지의 흐름을 처음 접하는 사용자가 5분 이내에 완료할 수 있다.
- **SC-004**: Get Started 가이드의 단계별 진행 화면에서 사용자의 진행 상태가 시각적으로 명확히 구분되어 전체 완료율을 즉시 파악할 수 있다.
- **SC-005**: 모든 주요 화면(대시보드, 문서 조회, Blueprint 생성, Get Started, 프로젝트 상세)이 단일 앱에서 일관된 레이아웃으로 제공된다.
- **SC-006**: 화면 전환 시 로딩 상태가 명확히 표시되어 사용자가 작업 진행 여부를 인지할 수 있다.
- **SC-007**: Empty State 화면이 모든 리스트/목록 화면에서 제공되어 사용자가 다음 행동을 즉시 이해할 수 있다.

## Assumptions

- 인증/로그인 기능은 범위 외이며, 사용자는 이미 로그인된 상태로 가정한다(Mock 사용자 고정).
- 모든 데이터는 Mock으로 구현되며, 실제 API 연동, GitHub API, Confluence API 통신은 이번 범위에 포함되지 않는다.
- UI는 데스크탑 우선(1280px 이상) 레이아웃으로 설계하며, 모바일 반응형은 v2에서 다룬다.
- 기존 `packages/ui`의 `@myorg/ui` 컴포넌트(DocsHubLayout 포함)를 재사용한다.
- 앱은 monorepo 내 `apps/ai-wiki-portal-platform-by-claude`로 신규 패키지를 추가하여 구성한다. pnpm workspace를 통해 `@myorg/ui`를 workspace 참조로 사용한다.
- 다국어(i18n)는 범위 외이며, UI 텍스트는 한국어로 작성한다.
- AI 실제 응답 기능은 구현하지 않으며, AI 도움말/질의 패널은 Mock 텍스트로 표현한다.
- Storybook의 `DocsHubLayout.stories.tsx`를 레이아웃 및 디자인 참고 기준으로 삼는다.
