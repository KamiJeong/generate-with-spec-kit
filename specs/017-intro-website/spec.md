# Feature Specification: 서비스 소개 웹사이트

**Feature Branch**: `017-intro-website`  
**Created**: 2026-04-14  
**Status**: Implemented  
**Input**: User description: "Build a lightweight introduction website for our service."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 첫 방문자의 서비스 가치 파악 (Priority: P1)

처음 방문한 사용자가 페이지를 열고 몇 초 안에 이 서비스가 무엇이고, 어떤 가치를 제공하는지 파악한다. 히어로 섹션의 헤드라인과 설명을 읽고 서비스의 핵심 가치 제안을 이해한다.

**Why this priority**: 방문자가 서비스를 이해하지 못하면 나머지 콘텐츠를 읽지 않고 이탈한다. 이것이 모든 다른 목표의 전제 조건이다.

**Independent Test**: 히어로 섹션만 구현하고, 5명의 첫 방문자에게 "이 서비스가 무엇을 하는지 한 문장으로 설명하라"고 요청해 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 첫 방문자가 랜딩 페이지에 도착했을 때, **When** 스크롤 없이 상단 화면을 본다면, **Then** 서비스 이름, 핵심 가치를 담은 헤드라인, 간략한 설명, 기본 CTA 버튼이 모두 보여야 한다.
2. **Given** 방문자가 히어로 섹션을 읽었을 때, **When** "이 서비스가 무엇인가요?"라는 질문을 받는다면, **Then** 방문자는 서비스의 핵심 기능을 한 문장으로 설명할 수 있어야 한다.
3. **Given** 모바일 기기 방문자가 페이지를 열었을 때, **When** 히어로 섹션을 본다면, **Then** 텍스트가 잘리지 않고 버튼이 탭하기 쉬운 크기로 표시되어야 한다.

---

### User Story 2 - 잠재 클라이언트의 신뢰 형성 (Priority: P2)

서비스를 평가 중인 잠재 클라이언트 또는 파트너가 페이지를 스크롤하며 핵심 강점, 검증 지표, 신뢰 근거를 확인한다. 충분한 신뢰를 형성한 후 문의 또는 인콰이어리 요청을 통해 다음 행동을 취한다.

**Why this priority**: 신뢰 형성은 잠재 클라이언트가 CTA를 클릭하도록 이끄는 핵심 동인이다.

**Independent Test**: 강점 섹션과 신뢰 섹션을 구현한 후, 평가자에게 "이 서비스를 신뢰할 이유를 찾았는가?"를 물어 확인할 수 있다.

**Acceptance Scenarios**:

1. **Given** 방문자가 페이지를 스크롤할 때, **When** 강점 섹션에 도달한다면, **Then** 3~6개의 핵심 강점이 스캔하기 쉬운 형태(아이콘 또는 카드)로 표시되어야 한다.
2. **Given** 방문자가 신뢰 섹션을 볼 때, **When** 지표, 클라이언트, 인증, 후기 중 하나 이상의 증거를 확인한다면, **Then** 방문자가 서비스의 신뢰성을 판단할 수 있는 구체적인 근거를 얻어야 한다.
3. **Given** 방문자가 CTA 섹션에 도달했을 때, **When** 다음 행동을 취하고 싶다면, **Then** 명확한 CTA 버튼 또는 링크가 즉시 보여야 한다.

---

### User Story 3 - 내부 이해관계자의 외부 공유 (Priority: P3)

내부 이해관계자가 외부 파트너나 잠재 고객에게 이 URL을 공유하며 서비스를 소개한다. 공유받은 사람은 링크를 열어 서비스의 정체성과 신뢰도를 빠르게 파악한다.

**Why this priority**: 외부 공유 시나리오는 P1과 P2 시나리오가 완성될 때 자동으로 충족된다.

**Independent Test**: 완성된 페이지 URL을 외부인에게 공유한 뒤 "이 서비스를 신뢰할 수 있겠는가?"를 확인한다.

**Acceptance Scenarios**:

1. **Given** 외부인이 공유 링크를 열었을 때, **When** 페이지를 처음 본다면, **Then** 회사/서비스 정체성이 헤드라인과 개요 섹션에서 명확히 드러나야 한다.
2. **Given** 방문자가 페이지 전체를 스크롤했을 때, **When** 첫 방문이 끝난다면, **Then** 서비스에 대해 연락하거나 더 알아볼 수 있는 경로가 최소 한 곳 이상 제공되어야 한다.

---

### Edge Cases

- 방문자가 JavaScript를 비활성화한 경우, 핵심 콘텐츠(히어로, 강점, CTA)는 여전히 표시되어야 한다.
- 매우 느린 네트워크 환경에서 이미지나 폰트가 늦게 로드될 때 레이아웃이 크게 깨지지 않아야 한다.
- 초소형 화면(320px)에서도 CTA 버튼이 잘리지 않아야 한다.
- 실제 마케팅 카피가 없는 상태에서도 섹션 구조가 명확해야 한다(플레이스홀더 텍스트 사용 가능).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-000**: 페이지는 로고, 섹션 앵커 링크, CTA 버튼을 포함하는 고정(sticky) 상단 내비게이션 헤더를 제공해야 한다. 헤더는 스크롤 시에도 항상 화면 상단에 유지되어야 한다. 모바일 화면에서는 섹션 링크가 햄버거 메뉴 아이콘으로 접혀야 하며, 아이콘 탭 시 드로어 또는 드롭다운으로 링크가 표시되어야 한다.
- **FR-001**: 페이지는 스크롤 없이 보이는 영역(above the fold)에 서비스 이름, 핵심 헤드라인, 보조 설명, 기본 CTA 버튼을 포함하는 히어로 섹션을 제공해야 한다. 배경 이미지를 사용하는 경우 저작권 문제 없는 자체 이미지로 교체되어야 한다.
- **FR-002**: 페이지는 서비스/회사가 무엇을 하는지 간략히 설명하는 개요 섹션을 포함해야 한다.
- **FR-003**: 페이지는 3~6개의 핵심 강점 또는 기능을 스캔 가능한 카드 또는 리스트 형태로 보여주는 강점 섹션을 포함해야 한다.
- **FR-004**: 페이지는 숫자 지표(예: 고객 수, 프로젝트 수)와 후기 인용구를 포함하는 신뢰 형성 섹션을 제공해야 한다.
- **FR-005**: 페이지는 방문자가 문의 또는 인콰이어리를 요청할 수 있도록 유도하는 최종 CTA 섹션을 포함해야 한다. CTA 섹션은 이름, 이메일, 메시지 필드를 포함하는 인라인 문의 폼을 제공해야 한다.
- **FR-006**: 모든 섹션은 모바일(320px 이상), 태블릿(768px 이상), 데스크톱(1024px 이상) 화면에서 올바르게 표시되어야 한다.
- **FR-007**: 페이지는 시맨틱 HTML 구조를 사용하여 스크린 리더와 검색 엔진이 콘텐츠를 올바르게 해석할 수 있어야 한다. 페이지는 WCAG 2.1 AA 기준을 준수해야 한다.
- **FR-008**: Storybook에 존재하는 UI 컴포넌트와 레이아웃 패턴을 우선적으로 재사용해야 한다. 새로운 프레젠테이션 컴포넌트는 기존 패턴이 충분하지 않을 때만 추가한다.
- **FR-009**: 기본 CTA는 단일하고 명확해야 하며 페이지 전체에서 일관된 행동(예: 문의하기)을 유도해야 한다.
- **FR-010**: 페이지는 실제 마케팅 카피가 없는 상태에서도 구조가 명확해야 하며, 나중에 실제 텍스트로 교체할 수 있도록 플레이스홀더 텍스트를 사용할 수 있다.

### Key Entities

- **내비게이션 헤더**: 로고, 섹션 앵커 링크, CTA 버튼을 포함하며 스크롤 시에도 화면 상단에 고정되는 헤더 영역. 모바일에서는 햄버거 메뉴 → 드로어/드롭다운 패턴으로 전환된다.
- **히어로 섹션**: 서비스 이름, 헤드라인, 보조 설명, CTA 버튼으로 구성된 페이지의 첫 번째 시각적 영역
- **개요 섹션**: 서비스/회사의 정체성과 제공 가치를 간략히 설명하는 섹션
- **강점 카드**: 핵심 강점 또는 기능 하나를 아이콘/제목/설명으로 표현하는 재사용 가능한 UI 단위
- **신뢰 근거**: 숫자 지표(고객 수, 프로젝트 수 등)와 후기 인용구로 구성되는 신뢰 형성 콘텐츠 항목
- **CTA 섹션**: 방문자가 다음 행동(문의, 인콰이어리)을 취하도록 유도하는 페이지 최하단의 독립 섹션. 이름, 이메일, 메시지 필드를 포함하는 인라인 문의 폼을 포함한다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 첫 방문자가 페이지를 열고 5초 이내에 서비스가 무엇인지 한 문장으로 설명할 수 있어야 한다.
- **SC-002**: 기본 CTA 버튼은 데스크톱, 태블릿, 모바일 모든 환경에서 스크롤 없이 또는 한 번의 스크롤 이내에 도달 가능해야 한다.
- **SC-003**: 페이지는 모바일, 태블릿, 데스크톱 세 가지 화면 크기에서 레이아웃 깨짐 없이 표시되어야 한다.
- **SC-004**: 페이지는 WCAG 2.1 AA 기준을 준수해야 하며, 스크린 리더 사용자가 모든 주요 섹션을 올바른 순서로 탐색할 수 있어야 한다.
- **SC-005**: 페이지 내 모든 섹션이 실제 카피 없이도 구조와 의도가 명확히 전달되어야 한다(플레이스홀더 텍스트로 검증 가능).
- **SC-006**: 시각적 스타일이 기존 Storybook 디자인 시스템과 일관성을 유지하여, 기존 제품을 아는 사용자가 동일한 브랜드임을 즉시 인식할 수 있어야 한다.

## Clarifications

### Session 2026-04-14

- Q: CTA를 클릭했을 때 어떤 동작이 발생해야 하는가? → A: 페이지 내 인라인 문의 폼 (이름, 이메일, 메시지 필드)
- Q: 준수해야 할 접근성 기준 수준은 무엇인가? → A: WCAG 2.1 AA
- Q: 페이지 상단에 내비게이션 헤더가 필요한가? → A: 고정(sticky) 상단 헤더 — 로고 + 섹션 앵커 링크 + CTA 버튼
- Q: 신뢰 섹션에 포함할 증거 유형은 무엇인가? → A: 숫자 지표(고객 수, 프로젝트 수 등) + 후기 인용구
- Q: 모바일에서 내비게이션 링크를 어떻게 처리해야 하는가? → A: 햄버거 메뉴 아이콘 → 드로어/드롭다운으로 링크 표시

## Assumptions

- 이 웹사이트는 단일 페이지(Single Page) 형태로 구현되며, 별도의 라우팅이나 다중 페이지 구조는 필요하지 않다.
- React 18 이상을 사용하며, React 19.x도 사용 가능하다.
- 실제 마케팅 카피(헤드라인 문구, 강점 내용, 신뢰 지표)는 나중에 삽입될 것이므로, 현재는 구조와 레이아웃에 집중한다.
- CTA 섹션은 이름, 이메일, 메시지 필드를 포함하는 페이지 내 인라인 문의 폼으로 구현한다. 폼 제출 후 동작(서버 전송, 이메일 연동 등)은 이번 범위에 포함되지 않으며 플레이스홀더 처리한다.
- 기존 Storybook에 등록된 UI 컴포넌트와 레이아웃 패턴이 충분히 다양하여 대부분의 섹션에서 재사용 가능하다고 가정한다.
- 사이트는 CMS, 백엔드 API, 데이터베이스 없이 정적 콘텐츠만으로 동작한다.
- 다국어 지원(i18n)은 이번 범위에 포함되지 않는다.
- 복잡한 진입 애니메이션이나 인터랙션 효과는 포함하지 않으며, 필요한 경우 단순한 전환 효과만 사용한다.
